# geosin.be — Honest Audit

**Date:** 10 June 2026
**Reviewed:** all 15 pages, `auth.js`, `style.css`, 3 SQL files, 2 Supabase edge functions, live site at https://geosin.be

---

## The one-paragraph verdict

This is genuinely impressive for a first website — it's live, bilingual, has real auth, a moderated forum, rate limiting, bot protection, and a clean design system. You've built something most "beginners" never get close to. But you have **one serious privacy bug that leaks every member's email address to anyone on the internet**, and underneath the polish there's a maintenance trap: the same nav, footer, and ~10 copies of the same helper code are pasted into every page by hand. Nothing here is fatal, but the email leak should be fixed today, and the copy-paste problem will slow every future change you make. Details and fixes below, worst-first.

## 🔴 Critical — fix today

### 1. Every member's email address is readable by anyone

This is the big one. In `SETUP.sql` the `profiles` table holds `email`, and the policy is:

```sql
create policy "Profiles are public"
  on profiles for select
  to anon using (true);
```

`using (true)` means *any visitor*, logged in or not, can open the browser console and run:

```js
sb.from('profiles').select('email')
```

…and download the email of every person who ever signed up. For an EU community site this is also a GDPR problem, not just a "bug." The same leak exists on the forum: `forum_threads` and `forum_replies` store `author_email`, and the "view approved threads/replies" policies return the whole row, so non-incognito posters' emails are exposed too.

**Fix:** stop exposing email. Two options — do at least the first:

```sql
-- 1. Make profiles expose only safe columns via a view, OR drop email from public reads.
--    Simplest: restrict the profiles SELECT policy so people only read their own row,
--    and create a separate public view with just display_name + incognito_name.
drop policy "Profiles are public" on profiles;

create policy "Users read own profile"
  on profiles for select to authenticated
  using (auth.uid() = id);

create view public_profiles as
  select id, display_name, incognito_name from profiles;
grant select on public_profiles to anon, authenticated;
```

Then change the forum/notification code that reads `profiles` to read `public_profiles` instead. For `author_email`, stop selecting it on the client (the forum render only needs `author_name`), and ideally split it into a separate admin-only table or column-level policy. **If you only do one thing from this whole document, do this.**

### 2. Open redirect on login

`login.html` does:

```js
location.href = new URLSearchParams(location.search).get('next') || 'account.html';
```

`next` comes straight from the URL with no validation. A link like `login.html?next=https://evil-phishing-site.com` will send a user there *after* they log in — a classic phishing trick that abuses your trusted domain.

**Fix:** only allow relative paths.

```js
const next = new URLSearchParams(location.search).get('next') || 'account.html';
const safe = (next.startsWith('/') || /^[\w-]+\.html/.test(next)) && !next.startsWith('//');
location.href = safe ? next : 'account.html';
```

## 🟠 High — fix this month

### 3. You repeat yourself everywhere (the maintenance trap)

The navbar, the footer, the `setLang()` function, the `escapeHtml` helper, the `timeAgo` helper, and the Supabase `<script>` tags are **hand-copied into all 15 HTML files**. I counted the `escapeHtml`/`escapeHTML` helper defined separately in at least 8 files — and it's even named two different ways (`escapeHtml` in forum/jobs/housing, `escapeHTML` in account). Right now, changing one nav link means editing 14 files and hoping you didn't miss one or introduce a typo.

This is the single biggest thing holding the project back, even though it's invisible to visitors. Every feature you add from here gets more expensive.

**Fix (pick one, easiest first):**
- **Easy:** move all shared JS (setLang, escapeHtml, timeAgo, FAQ toggle, mobile nav) out of the page `<script>` blocks and into `auth.js` (or a new `shared.js`) which is already loaded everywhere. Delete the duplicated copies.
- **Medium:** inject the nav + footer with one JS function instead of pasting the HTML. One `renderNav()` in `auth.js` that writes the markup into a `<div id="nav"></div>`.
- **Proper:** adopt a tiny static-site setup (Astro, Eleventy, or even simple HTML includes) so nav/footer live in one file. Bigger learning curve, but it's the real answer and a great next thing to learn.

### 4. "Privacy" link goes nowhere — and you have no privacy policy

The footer's `Privacy` link is `href="#"` and there is no privacy page. Given that you collect emails and run on EU users, you legally need a privacy policy (what you collect, why, how to delete it). This pairs directly with issue #1.

**Fix:** create `privacy.html` (I can draft one), link it from every footer, and add it to the sitemap.

### 5. The homepage's live numbers can sit broken silently

On the live site right now the forum stats show "—" and "Recent activity" shows "Loading…" — they never resolved during my fetch. If Supabase is slow or a query errors, `loadForumStats()` and `loadSpotlight()` fail quietly with no fallback text, so the homepage looks half-broken to a visitor. (This may just be because the forum is empty, but the code doesn't visibly distinguish "empty" from "error" for the stats.)

**Fix:** wrap those loaders in try/catch and always render a sensible default ("Be the first to post"), and set the stat numbers to `0` rather than leaving `—` if the query returns.

## 🟡 Medium — worth doing

### 6. An 865 KB image ships that nothing uses
`assets/pexels-beata-makuch-76906295-17352442.jpg` (865 KB) isn't referenced by any page or stylesheet. It's dead weight in the repo. The hero image (`hero-bruges.jpg`, 314 KB) *is* used but could be ~100 KB as WebP. **Fix:** delete the pexels file; convert the hero to WebP with a JPG fallback.

### 7. The hero alt text is wrong in one language and the photo/caption disagree with the comment
The hero `alt` says "Canal houses in Bruges," the visible heading says "Belgium," but the code comment and fallbacks reference Brussels' Grand-Place. Minor, but the alt text should match the actual image, and the comment is now misleading. **Fix:** confirm which photo you're using and align the alt + comment.

### 8. SEO basics are inconsistent
`events.html` has **two** `<meta name="description">` tags; `account.html`, `admin.html`, `404.html`, `reset-password.html` have **none**. Only the homepage has Open Graph image tags missing entirely (no `og:image` anywhere), so links shared on Facebook/WhatsApp show no preview image — a real loss for a community that lives on Facebook. **Fix:** one unique description per page, remove the duplicate, add a single `og:image` (your logo or hero) site-wide.

### 9. Both languages are always in the DOM
Your i18n works by rendering EN *and* KA into every page and hiding one with JS/CSS. It's simple and it works, but: (a) screen readers and search engines see both languages mashed together ("Servicesსერვისები"), (b) there's a brief flash of both before JS runs on slow connections, and (c) it roughly doubles your HTML size. For now it's an acceptable trade-off given your skill level — just know it's a real limitation, not a best practice. **Fix (later):** `lang` attributes + `hreflang`, or separate `/en/` and `/ka/` pages when you move to a static-site generator (#3).

### 10. `getSession()` lock is bypassed globally
In `auth.js` you disabled the Supabase Web Locks coordination (`lock: (_n,_t,fn) => fn()`) to dodge a hang. Reasonable workaround for a small site, but it can cause two tabs to race on token refresh and occasionally log a user out. **Fix:** leave it for now, but add a comment with the date and revisit if users report random logouts.

## 🟢 Low / polish

- **`404.html` is 14 KB and 298 lines** — it inlines the entire nav/footer/styles. Fine, but it's another copy to maintain (see #3).
- **`.cursorrules`, `NEXT_STEPS.md`, `ROADMAP.md`, `TRANSLATE_ME.md`, `DESIGN_BRIEF*.md`** are shipping in the repo. They're excluded from the Cloudflare deploy via `.assetsignore` (good), so this is cosmetic.
- **Copyright says "© 2025"** across the footer; it's mid-2026. **Fix:** make it dynamic with `new Date().getFullYear()`.
- **`escapeHtml` naming inconsistency** (`escapeHtml` vs `escapeHTML`) — harmless today but exactly the kind of thing that bites you when you consolidate (#3). Pick one name.
- **No analytics / error tracking.** You're flying blind on what visitors actually do or when something breaks. A privacy-friendly option (Plausible, Cloudflare Web Analytics) is a nice low-effort add.

## What's genuinely good (don't lose this)

- **Security architecture is mostly right.** RLS is enabled on every table, the publishable key in `auth.js` is the *correct* key to expose publicly (security rests on RLS, not secrecy), Turnstile + IP rate limiting + per-user post limits are all real, server-side, and well thought out. This is better than a lot of professional sites.
- **The `escapeHtml`-everywhere habit** means you've actually defended against XSS in your user-generated content (forum, listings). Good instinct.
- **Admin panel is properly gated** server-side via the `is_admin()` RLS function, not just hidden in the UI.
- **The design system** (CSS variables, consistent tokens, the bento/journey/trust sections) is coherent and genuinely nice-looking.
- **Thoughtful UX touches:** cached auth state to avoid nav flicker, incognito forum names, progress saved locally, graceful image fallbacks.
